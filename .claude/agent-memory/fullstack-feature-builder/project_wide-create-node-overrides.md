Wide Creation (Step 2 "Structure") supports optional per-item field overrides that beat Step 3 bulk config.

**Decision:** Overrides live in a separate `nodeOverrides: Record<nodeId, fields>` map in `useWideCreationStore`, NOT in each node's `fields`. They are merged on top of `node.fields` only at export time inside `toTemplate` (item-level wins because it spreads last). `nodeIdFor(ci, ai?, di?)` produces position-based ids ("c0", "c0:a1", "c0:a1:d0") using GLOBAL campaign index within `store.campaigns` (StepStructure builds a campaignIdâ†’globalIndex map since the tree renders grouped by objective with per-objective indexes).

**Why:** Step 3 bulk config writes into `node.fields` directly via `bulkUpdate*`. Keeping overrides in a parallel map lets the indicator/clear UI distinguish item-level edits from bulk values, and makes "clear overrides" trivial. `generateStructure` and `reset` both wipe `nodeOverrides` because position ids don't survive a structural regenerate.

**How to apply:** The backend (`WideCreationService.generateFromTemplate`) already prioritizes explicit `fields.name` over the naming pattern (`resolvedFields.name || resolveNamingPattern(...)`) at all three levels and `resolveInheritedFields` preserves a node's own fields â€” so per-item overrides flowed through with ZERO backend change. The ad-level creative override must be stored as `creative: { creative_id }` (object), not a flat `creative_id` key, because the service only reads `resolvedAdFields.creative`. `setNodeOverride` strips ''/null/undefined values so clearing a field reverts to the bulk default. See [[arrayfield-react-key-focus]] â€” override inputs key by stable nodeId/field.key, so no focus-loss risk.

**Override panel field coverage (StepStructure.tsx, after 2026-05-27 expansion):** Panels are a self-contained lightweight field-descriptor system (`OverrideFieldDef[]` + `OverrideField` renderer) â€” NOT the schema engine. Field types: text/number/datetime/select/multiselect/boolean(tri-state select)/json/creativeId/trackingSpecs. Campaign: name,status,special_ad_categories(multiselect),buying_type,daily_budget,lifetime_budget,bid_strategy,bid_amount. AdSet: name,status,optimization_goal,destination_type,billing_event,daily_budget,lifetime_budget,bid_strategy,bid_amount,start_time,end_time,is_dynamic_creative,targeting(json),promoted_object(json). Ad: name,status,creative,url_parameters,tracking_specs(trackingSpecs â€” structured list editor, NOT JSON textarea as of 2026-05-27; component is `frontend/src/components/meta/TrackingSpecsEditor.tsx`, exports `TrackingSpec` type; stores `TrackingSpec[]`, emits `undefined` on empty array so store drops the override). Gotchas for future edits: (1) enum option lists are duplicated from MetaFieldRegistry â€” registry is source of truth, keep in sync (special_ad_categories real enum is NONE/EMPLOYMENT/HOUSING/ISSUES_ELECTIONS_POLITICS/FINANCIAL_PRODUCTS_SERVICES/ONLINE_GAMBLING_AND_GAMING, NOT the CREDIT list some specs claim). (2) bid_amount uses `showWhen: requiresBidAmount` (only shows for BID_CAP strategies LOWEST_COST_WITH_BID_CAP/COST_CAP/LOWEST_COST_WITH_MIN_ROAS). (3) multiselect must emit `undefined` when empty (store only strips ''/null/undefined, NOT []). (4) boolean is a tri-state select (blank=bulk default) because `false` is meaningful and a checkbox can't express "no override". (5) json fields (`JsonOverrideField`) hold raw text in local useState, push parsed object up only when valid, useEffect re-syncs on external value change (e.g. Clear overrides). Backend `generateFromTemplate` consumes most fields via resolved*Fields, but does NOT currently map adset `is_dynamic_creative` or campaign `bid_amount` to the Meta payload, and `status` is always forced PAUSED â€” those overrides persist in draft fields JSON only.
