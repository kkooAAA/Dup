# Draft System

The draft system allows users to import Meta campaigns into a local PostgreSQL database, edit them with dynamic forms, validate against Meta constraints, and publish back to Meta.

## Entity Hierarchy

```
DraftCampaign
├── DraftAdSet
│   ├── DraftAd
│   └── DraftAd
├── DraftAdSet
│   └── DraftAd
```

Each entity has:
- `id` — local UUID
- `metaId` — corresponding Meta object ID (null until published, or set if imported from existing)
- `userId` — owner
- `adAccountId` — Meta ad account
- `name` — display name
- `data` — JSON blob of all Meta fields
- `status` — `DRAFT` | `PUBLISHING` | `PUBLISHED` | `FAILED`

## Draft Lifecycle

1. **Import** — `DraftService.duplicateCampaignToDraft()` fetches a full campaign tree from Meta and stores it locally
2. **Edit** — Users modify fields via dynamically-generated forms. The form schema is generated by `MetaFormSchemaEngine` based on the current objective and field definitions from [[meta-field-registry]]
3. **Validate** — `DraftValidationEngine.validateFullDraft()` checks all fields against Meta constraints before publishing
4. **Publish** — `DraftPublishService.publishCampaign()` creates campaign → adsets → ads on Meta sequentially, updating local records with Meta IDs

## Conversion via Drafts

When converting objectives, `DraftService.convertCampaignToDraft()` uses `ObjectiveConversionService` to transform the campaign data to the target objective, then stores the result as a draft. This allows the user to review and edit the converted structure before publishing.
